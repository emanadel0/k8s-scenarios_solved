scenario 9 explanation why hpa is the best solution::

The application receives unpredictable traffic spikes, which means we can't know in advance how many pods we need. Manually changing replicas would be too slow and requires human intervention 24/7 and this is not practical... 

So HPA was the right choice because:
1. It's automatic — no human needed. When traffic spikes, HPA detects the CPU rising and adds pods by itself. When traffic drops, it removes them.
2. CPU was the right metric — this is a web application. More requests = more processing = higher CPU. CPU is a direct reflection of real load on this app.
3. It's cost-efficient — we set minReplicas: 1 and maxReplicas: 5. The cluster never wastes resources running 5 pods when traffic is low, and never struggles with 1 pod when traffic is high.
4. The 50% CPU threshold is a safe middle ground — pods start scaling up before they're fully overwhelmed, giving new pods time to start before performance degrade
